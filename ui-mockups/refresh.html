NimbusImage · UI Refresh · v0.2

Let the image breathe.
Let the chrome dissolve.

A staged refresh that retires the heavy side panels in favour of a modular workspace of floating glass palettes. The app bar goes transparent. Tools and channels live on circular icon clusters that hover above the image. The interface is something you reach for — not something you wade through.

01 · Direction

Four ideas that guide every decision.

Before any pixel moves, agreement on principles. Every component in the refresh should defend at least one of these. If it defends none, it shouldn't ship.

Image is the hero

Chrome retreats. Translucency, blur, and edge-only borders ensure pixels are never wasted on opaque panels covering the sample.

P · 01

Recombinant by default

Each panel becomes a palette: draggable, dockable, dismissible. The scientist chooses which tools are visible at a given moment.

P · 02

Single accent, used quietly

Teal stays as the one true accent — but only for state. Selection. Active. Confirmation. Everything else is greyscale and typography.

P · 03

Nothing is lost

Every existing feature has a place. The refresh is composition, not amputation. Power users keep their density; new users see less at once.

P · 04
02 · North Star

The final state — everything floats.

A vertical tool dock on the left. A bottom dock for navigation. Three icon-button clusters in the app bar, each opening a glass palette that hovers exactly where you summoned it. The image runs edge to edge. The chrome is there when you reach for it, gone when you don't.

02b · Surfaces

Four surfaces. Each with one job.

Not every panel deserves the same surface. The refresh introduces four distinct surface types. When a feature lands on the wrong surface the workspace feels noisy; when the surface matches the intent the interaction disappears. The job of design here is matchmaking.

A · Dock

The dock

Always-visible icon strip pinned to an edge. Single-select. Long-press reveals sub-options.

  • Drawing tools — pointer, blob, polygon, point, line
  • Interaction tools — connect, SAM, worker
  • Z navigation + view controls (bottom)
Rule

Use a dock when the user switches between modes constantly. Muscle memory is the win.

B · Palette

The palette

Floating glass panel. Toggleable, draggable, dismissible. Stays open while the user works.

  • Layers / Channels
  • Object Browser · Properties
  • Tool Inspector · contextual
  • Chat / AI · Snapshots (jump-to)
Rule

Use a palette when the user wants to keep it open while doing other work on the canvas.

C · Popover

The popover

Small transient menu near its trigger. Opens, user picks, closes. No persistent UI footprint.

  • Dataset switcher
  • Tool sub-options (long-press dock)
  • Account menu · storage status
  • "More actions" overflow menus
Rule

Use a popover when the choice is brief and ephemeral. No commitment, no canvas overlap.

D · Drawer

The drawer

Large overlay sheet from the right edge. The user commits to this task; the canvas dims behind blur.

  • Upload data · Add new tool
  • Settings · Help docs
  • Export · Project management
  • Snapshots (manage view)
Rule

Use a drawer when the task IS the workflow — not a side activity that runs alongside the canvas.

Where each surface lives

Each surface gets its own zone in the chrome. The user can tell what type of action a button represents from where it lives: left edge = tool · right edge = panel · top corners = app-level.

Complex tools get a contextual inspector.

The dock just selects the tool. Selecting opens (or updates) a Tool Inspector palette next to it. The inspector adapts to the active tool — one palette, many configurations. This is the same relationship Figma has between its top-bar tools and its right-side properties panel.

04 · Roadmap

Four phases. Each shippable on its own.

The refresh is large, but the steps aren't. Each phase delivers a self-contained, mergeable improvement. If we stop after phase 2, the app is already lighter and more modern. If we go all the way, we have a fully modular workspace.

Phase 01 · Visual

The bar goes transparent.

Lowest-risk, highest-impact starter. The app bar gets a glass background — backdrop-filter blur + linear gradient — and the image starts immediately at top: 0. Buttons keep their current behaviour; only their container changes.

Effort ~ 1 day
Risk Low
Scope CSS only
Reversible Trivially
  • Change .v-app-bar background to glass with backdrop-filter
  • Make ImageViewer fill viewport edge-to-edge — adjust z-index stack
  • Add subtle bottom border gradient (1px) so the bar still reads as a region
  • Verify legibility against worst-case bright fluorescence images — add text-shadow fallback if needed
  • Audit overflow + tooltip behaviour now that bar is over image content
  • QA on Safari (backdrop-filter quirks) and on systems without GPU compositing

The big move. Left "Toolset / Layers" and right "Object Browser" become palettes. The left panel splits into two: a vertical tool dock (always visible, like Figma) and a Layers palette (toggleable). The right panel becomes a single Object Browser palette with everything its current incarnation has.

05 · Notes

Risks, decisions, and open questions.

Technical considerations

  • Backdrop-filter performance. Stacking many blurred surfaces (bar + tooldock + 2 palettes + bottomdock) can tax low-end GPUs. We should profile early. Mitigation: limit to ~5 concurrent blurred elements; pin Safari behind a feature query.
  • Vuetify 4 fit. v-app-bar's flat + color="transparent" + border="b" won't quite get us there. Likely we shadow the bar with a custom layout component and skip v-app-bar's scroll-behaviour logic.
  • Workspace persistence. Palette positions are personal, not shared — they belong in IDatasetView or a new userWorkspace resource, not IDatasetConfiguration.
  • Legibility over bright images. Some fluorescence samples are dominated by bright greens or saturated reds. The bar's tint may need to adapt — consider luminance-aware backdrop tinting or a fixed minimum 40% dark gradient on the bar's top edge.
  • Migration path. Recommend shipping Phase 3 behind a "New workspace" toggle in Settings. Keep the classic layout for ~2 releases. This gives the field time and protects power-user muscle memory.
  • Image viewer z-index. Today the canvas is bounded by panels. When palettes float, the canvas must extend behind them. GeoJS handles this fine, but mouse-event hit-testing for the canvas needs to be confirmed when palettes are over it (pointer-events: none on palette gaps).
  • Tooltip strategy. Circular icons need consistent tooltips with shortcuts. Standardize on a single tooltip component now to avoid drift later.
  • Accessibility floor. All palette behaviour must work via keyboard. Focus must return to the originating icon button on close. Pin/collapse must be reachable without a mouse.

Open questions for you

  • Always-float, or pin-pushes-canvas? Lightroom pins panels and shrinks the image. Figma floats panels above the canvas. I'm leaning Figma — does that match your instinct?
  • Tool dock: always visible, or summonable? If always visible, scientists never lose their drawing tool. If summonable, the image is fully unobstructed. Recommend always-visible for the dock, summonable for everything else.
  • Workspace presets in v1 or v2? Worth the effort, but might be deferrable to a follow-up. Your call.
  • Type direction. Geist + Instrument Serif is one option. If you'd rather stay closer to the current Roboto/system, we can keep that and only refresh weights and sizes.
